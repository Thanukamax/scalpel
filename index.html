Scalpel

A Proton for Android. Make stock Wine / Box64 / DXVK / Turnip run x86 Windows games faster by injecting targeted fixes — without forking any of them.

The surgeon that operates on the emulation stack without removing the organ.

The number that reframed it

It started as a crash-out: a day spent trying to make a phone run a 2017 PC game faster, watching every config idea die on contact with reality. Then one measurement on NieR: Automata, on a flagship S24 Ultra:

Adreno 750 — 99.8% busy · 37% clock · 38 fps
≈ 10× the work native hardware would do for the same frame.

The bottleneck isn't a setting you forgot to flip. It's structural translation overhead baked into the emulation stack. I swept five config levers — GOS, Turnip version, resolution, env-flags, DXVK version — all flat, regressive, or wouldn't launch. Config knobs don't move it.

The principle

Inject, don't fork

Forking Wine, Box64, DXVK, and Turnip means owning millions of lines and losing every upstream fix. Config tuning doesn't move the needle. The third path is the one Valve's Proton already walks on desktop — and nobody walks on Android yet.

  1. Dependencies stay stock and auto-updating.
  2. Fixes ride on top through sanctioned injection points — not hacks.
  3. We maintain only the injection layer + a minimal, auto-rebased patch-overlay.
  4. Update = update the app. Never the deps.
